# Changelog

All notable changes to AI-CodeGuard are documented here. The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Added

- **`CG-021` Weak Cryptography now also flags the ECB block-cipher mode** (CWE-327), a common miss where identical plaintext blocks encrypt to identical ciphertext (the "ECB penguin"). One language-agnostic check covers every ecosystem's spelling of the mode — `aes-256-ecb` (Node/PHP algorithm strings), `AES/ECB/...` (Java transformation), `MODE_ECB` (pycryptodome) — gated to real cipher calls (`createCipheriv`, `Cipher.getInstance`, `AES.new`, `openssl_encrypt`, …) so an incidental `ecb` token elsewhere can't trip it. Authenticated/secure modes (GCM, CBC) are not flagged. Also flags Node's deprecated `crypto.createCipher`/`createDecipher` (no `iv`) — they derive the key from a password with a single unsalted MD5, a weakness regardless of algorithm; the secure `createCipheriv`/`createDecipheriv` replacements are matched by exact name and not flagged. Also catches Python's generic `hashlib.new('md5', …)` constructor form (previously only the direct `hashlib.md5(…)` was caught). 12 new tests. No new rule or ID — this deepens the existing `CG-021`.
- **`CG-050` Security Misconfiguration gains three patterns**: a raw wildcard `Access-Control-Allow-Origin: *` response header set via `res.setHeader(...)` / PHP `header(...)` (the existing `cors(...)`-keyed pattern only saw the middleware form; a specific origin is not flagged), and Python's two explicit ways to disable TLS certificate validation — `ssl._create_unverified_context()` and `ssl.CERT_NONE`. 5 new tests. Deepens the existing `CG-050`.
- **`CG-003` Code Injection now covers Python's built-in `exec(...)`** — arbitrary code execution just like `eval`, and previously missed. Gated to Python and to the bare builtin (no receiver), so `child_process.exec` (JS) and `exec()` (PHP) — which are *command* execution, CG-002's job — are not misclassified as code injection. 3 new tests. Deepens the existing `CG-003`.
- **`CG-041` Insecure Deserialization now covers Python's `dill` and `cloudpickle`** — pickle-based libraries common in ML/data pipelines (and Spark/Dask), with the same arbitrary-code-execution risk as `pickle`. `json.loads` and other safe deserializers are unaffected. 3 new tests. Deepens the existing `CG-041`.
- **`CG-031` Arbitrary File Read/Write file-sink coverage**: now also flags Node's `appendFile`/`appendFileSync` (append) and `unlink`/`unlinkSync` (delete) when the path is user-controlled — the same arbitrary-file-access risk as read/write. A static path is not flagged. 3 new tests. Deepens the existing `CG-031`.

- **Inline suppression comments.** A finding can be silenced with a `codeguard-ignore` directive in a comment on the same line, or `codeguard-ignore-next-line` on the line above (any language's comment syntax works, since the whole line is scanned). A bare directive suppresses every rule for the target line; `codeguard-ignore CG-001, CG-010` scopes it to the listed rules, and any trailing prose is treated as a free-text reason. Rule IDs are matched case-insensitively. Suppression runs per file at Stage 1, so a suppressed finding never reaches Stage 2 (no wasted LLM calls). The two directives are kept distinct (a same-line `codeguard-ignore` never leaks onto the next line) via a negative lookahead. New `src/scanner/suppression.ts`.
  - Suppressions are **not silent**: the scan reports how many findings were suppressed (`scan.suppressed` in JSON, a dimmed summary line in text). Run `scan --no-inline-suppression` to ignore the directives and report everything, so a reviewer can audit what's being hidden.
  - 14 new tests (11 unit + 3 end-to-end scanner integration, covering the count and the audit toggle).
- **`scan --fail-on <level>` controls the CI exit code.** Previously `scan` always exited non-zero when any high/critical finding was reported, with no override. It now accepts `--fail-on low|medium|high|critical|none`: the process exits `1` when a reported finding is at or above the threshold, and `--fail-on none` never fails the build (report-only mode). The default is `high`, preserving the previous behavior exactly. The shared `SEVERITY_RANK` / `meetsSeverity` helper (previously private to the scanner, used for `--severity` filtering) now lives next to the `Severity` type so the gate and the filter share one ranking. 8 new tests (`meetsSeverity` + `shouldFailBuild`).
- **JSON output now includes a machine-readable severity breakdown.** The `scan` summary object gains `totalFindings` and `severityCounts` (`{ critical, high, medium, low }`), mirroring the counts the text report already prints — so CI gates and dashboards can read a severity tally without re-aggregating the `findings` array. 1 new reporter test.

- **CWE labels are now surfaced in every output format, not just SARIF.** The JSON reporter adds a `cwe` field per finding (`"CWE-89"`, or `null` for a custom rule with no mapping) plus a `cweUri` linking to the MITRE definition page (matching what SARIF's `helpUri` already provides), and the text reporter shows the CWE next to the rule ID in each finding header (`CG-001 CWE-89  SQL Injection`). Both read from the same `src/rules/cwe.ts` map that already backs the SARIF tags, so the three formats stay consistent. 3 new reporter tests.
- **PHP rule depth: 6 → 11 rules**, closing the gap with Go/Java:
  - `CG-021` Weak Cryptography — bare `md5()`/`sha1()` calls (no algorithm argument needed — calling them at all is the signal) and `hash()` called with a weak algorithm string; `hash('sha256', ...)` is not flagged
  - `CG-040` Sensitive Data Exposure — `error_log`/`syslog` bare calls, plus `log`/`logger`-receiver method calls (`Log::error(...)` Laravel-style facades, `$logger->info(...)` Monolog-style) carrying password/token/secret/PII text
  - `CG-041` Insecure Deserialization — PHP's `unserialize()`, the classic PHP object-injection gadget-chain vector, reuses the existing JS `deserialize`/`unserialize` name check with no new matching code
  - `CG-050` Security Misconfiguration — `curl_setopt($ch, CURLOPT_SSL_VERIFYPEER, false)`/`CURLOPT_SSL_VERIFYHOST` and `ini_set('display_errors', 1)`/php.ini's `display_errors = 1`
  - 13 new tests plus a `stretch.php` vulnerable fixture (and safe counterparts in `secure-code.php`) covering all four
- **`CG-031` Arbitrary File Read/Write extended to Go, Java, and PHP** (previously JS/TS/Python only): Go's `os.Open/OpenFile/Create/ReadFile/WriteFile` combined with `r.URL.Query`/`r.FormValue`/`mux.Vars`/`os.Args`; Java's `new File(...)` or `Files`/`Paths` static helpers combined with `getParameter`/`getHeader`/`getQueryString`; PHP's `file_get_contents`/`file_put_contents`/`fopen`/`readfile` combined with `$_GET`/`$_POST`/`$_REQUEST`/`$_COOKIE`. 6 new tests.
- **`CG-010` XSS extended to Python and Java** (previously JS/TS only): Python's `mark_safe`(Django)/`Markup`(Flask/Jinja2's markupsafe)/`render_template_string` calls, and Java's `response.getWriter().write/print/println`. Both require a non-literal argument (a lone static string literal is inert output and is not flagged) to avoid flagging the common safe pattern of wrapping a fixed string as pre-escaped HTML. 6 new tests plus fixture coverage in `injection.py`/`secure-code.py` and `Stretch.java`/`SecureStretch.java`.

### Fixed

Caught by an 8-angle code review of the additions above, before any release:

- **`CG-031`'s Go `USER_INPUT_GO` regex missed the common quoted-argument form** — `\bc\.Param\(\b` required a word character immediately after the literal `(`, which a quoted string argument never satisfies (`(` and `"` are both non-word characters, so no `\b` boundary exists between them). `os.ReadFile(c.Param("filename"))` — the textbook Gin path-traversal-to-arbitrary-file-read pattern — silently failed to match; only the unrealistic unquoted form `c.Param(id)` did. Fixed by moving the anchor to after the method name instead of after the paren.
- **`CG-041` flagged PHP's `Serializable` interface method as the dangerous global `unserialize()`** — the shared JS/PHP name check (`DESER_FUNCTIONS_JS.includes(call.name)`) matched by name only, so `$session->unserialize($data)` (a class implementing the standard `Serializable` interface, an unrelated and benign OOP pattern) was flagged identically to the actual gadget-chain vector, PHP's bare global `unserialize($data)`. Now gated to `call.object === null` for PHP specifically.
- **`CG-040`'s PHP bare-function check was case-sensitive** — unlike the sibling Go/Java branches, `LOG_FUNCTIONS_PHP.includes(call.name)` compared raw text, so `Error_Log(...)`/`SYSLOG(...)` (valid, case-insensitive PHP) went undetected. Now lowercases `call.name` before comparing — safe for PHP (unlike JS, there's no `new Error(...)`-style collision risk to protect against).
- **`CG-010`'s Java sink check missed the standard two-step `PrintWriter` idiom** — requiring `"getWriter"` to appear in the same call expression as the sink method caught only the chained one-liner `response.getWriter().write(...)`, missing the far more common `PrintWriter out = response.getWriter(); out.println(...)` pattern where the receiver is a plain variable. Now also matches when the receiver was textually assigned from a `getWriter()` call in the surrounding context.
- **`CG-010`'s `hasOnlyStaticStringArg` treated a zero-argument call as dynamic** — `response.getWriter().println()` (inert; prints nothing but a newline) failed the "sole static string literal" regex against an empty argument list and was incorrectly flagged. Empty arguments are now treated as static.
- 5 new regression tests covering all five fixes (365 tests total).
- **New rule: `CG-022` Insecure Randomness (CWE-330)**, covering all 6 supported languages — a new rule category, not just a language extension. Flags a non-cryptographic PRNG (`Math.random`, Python's `random` module, Go's `math/rand`, `java.util.Random`, PHP's `rand`/`mt_rand`) used to build a token, session ID, password, or similar. Since Stage 1 has no dataflow analysis, "security-sensitive" is inferred from keywords (`token`/`session`/`password`/`secret`/`otp`/`api_key`/`reset`/`nonce`/`csrf`) in the surrounding source lines rather than tracing where the value flows — deliberately without `\b` word-boundary anchors, since the keyword is typically embedded in a camelCase/PascalCase identifier (`generateSessionID`) where a boundary never appears between the words. The cryptographic alternatives (`crypto.randomBytes`, Python's `secrets`, `crypto/rand`, `SecureRandom`, PHP's `random_bytes`/`random_int`) are not flagged, and non-security uses of the same insecure APIs (UI jitter, sampling, dice rolls) are left alone since they don't match the context heuristic. 10 new tests plus fixture coverage across all 6 languages. Built-in rule count: 13 → 14.
- **`RuleCheckContext.wasAssignedFrom(varName, sourcePattern, node, lines?)`** — a reusable, tested utility for the common "the value was assigned to a variable a few lines earlier, not passed inline" pattern. Looks for `varName = <rhs>` (or Go's `:=`) in the surrounding lines and tests `sourcePattern` only against that assignment's right-hand side (not the whole context window, so an unrelated occurrence of the pattern elsewhere nearby can't produce a false match). This is a textual approximation, not real dataflow analysis — Stage 1's parser produces a flat list of interesting nodes rather than a nested AST with function scoping (see `docs/design/RULES.md` §4), so there's no variable shadowing awareness and no multi-hop tracing.
  - `CG-031` (arbitrary file access) now catches the common two-step form across Go/Java/PHP/JS/TS/Python (e.g. Go's `path := r.URL.Query().Get("path"); os.Open(path)`), not just the path passed inline to the sink call. 6 new tests.
  - `CG-010`'s Java two-step `PrintWriter` detection (added in the fix above) now goes through this shared utility instead of a one-off regex, removing the duplication.
- **New rule: `CG-023` Insecure Regular Expression (ReDoS, CWE-1333)**, covering all 6 languages — another new rule category. Flags regex patterns with nested/overlapping quantifiers (`(a+)+`, `(a*)*`), the classic catastrophic-backtracking shape, in `new RegExp(...)` (JS/TS), `re.compile`/`re.match`/etc. (Python, gated to the `re.` receiver so it doesn't collide with unrelated `.split()`/`.match()` calls on strings), `regexp.MustCompile`/`Compile`/`MatchString` (Go), `Pattern.compile` (Java), and `preg_match`/`preg_match_all`/`preg_replace`/`preg_split` (PHP). This is a syntactic heuristic on the regex text — one common real-world shape, not a full NFA/backtracking-complexity analysis. 10 new tests plus fixture coverage across all 6 languages. Built-in rule count: 14 → 15.
- **New rule: `CG-024` NoSQL Injection (CWE-943)**, covering JS/TS, Python, and PHP — a third new rule category, distinct from `CG-001` SQL injection since it's about object *structure* (query operators), not string concatenation. Flags a MongoDB CRUD method (`find`/`findOne`/`updateOne`/`deleteOne`/etc.; pymongo's snake_case equivalents on the Python side — `find_one`/`update_one`/etc., since only `find`/`aggregate` are spelled the same as the JS/PHP camelCase driver APIs) called with the *entire* request object as the filter or update/replacement document (`users.find(req.body)`, `users.updateOne({_id}, req.body)`), which lets an attacker submit query operators (`$ne`, `$gt`, `$set`, ...) instead of a plain value and bypass checks like a password comparison or inject arbitrary fields via mass assignment. A specific field access (`req.body.username`) is an ordinary string value and isn't flagged — only the whole object being passed directly is the signal. Also flags a `$where` clause built with string concatenation/template literals, which MongoDB evaluates as arbitrary JavaScript (the NoSQL analogue of `CG-003`) — scoped to the `$where` key's own value so an unrelated dynamic sibling field elsewhere in the same call doesn't trigger it. `find`/`find_one` alone collide with unrelated common APIs (`Array.prototype.find`, Python's `str.find`), so a bare match on those two names gets reduced confidence instead of being suppressed outright. Not ported to Go/Java: their MongoDB driver APIs are more strongly typed and don't have a matching "pass a raw object directly" call shape. 10 new tests plus fixture coverage across all 3 languages. Built-in rule count: 15 → 16.
- **New rule: `CG-025` Open Redirect (CWE-601)**, covering all 6 languages — a fourth new rule category. Flags an HTTP redirect whose target is built from unvalidated user input: `res`/`response`/`reply`/`ctx` `.redirect(...)` (JS/TS), Flask/Django's bare `redirect`/`HttpResponseRedirect`/`HttpResponsePermanentRedirect` (Python), `http.Redirect(...)` (Go), `response.sendRedirect(...)` (Java, an unambiguous method name that needs no receiver check), and `header(...)` calls that actually set a `Location:` header (PHP — `header()` is used for far more than redirects, so the `Location:` check is required to avoid flagging unrelated uses). Written directly with the per-language lookup-table shape from the "Changed" section below rather than an if/else chain, since it was added after that consolidation. 12 new tests plus fixture coverage across all 6 languages. Built-in rule count: 16 → 17.

Caught by an 8-angle code review of the CG-024/CG-025 additions above, before merge:

- **JS/TS and Python's `extractCallInfo` mis-parsed chained calls** — it took the *first* `(` in the call text (`text.indexOf('(')`) instead of the outer call's own paren, so `res.status(302).redirect(url)` resolved to `name: "status", object: "res"` and `db.collection('users').find(query)` resolved to `name: "collection", object: "db"`. Every rule matching on `call.name`/`call.object` for these two languages silently missed chained-call call sites. Go/Java/PHP's adapters already used the backward-matching `findOuterArgumentsStart` helper for exactly this reason; JS/TS and Python now do too.
- **A non-identifier "receiver" could pattern-match unrelated rule keywords by coincidence** — fixing the chained-call bug above means `object` is now computed for *any* dot-preceded call, including a regex/string/array literal directly followed by a method call (`/\brequest\./.test(text)`, `"a,b".join(x)`). Several rules substring-match `object` against known receiver names, so a leftover literal happening to contain a matched keyword (e.g. the literal text `request` inside a regex pattern) produced a false hit — caught by this repo's own self-scan flagging `CG-060` on `ssrf.ts`'s own regex constant. `object` is now `null` unless it starts with a valid identifier character.
- **`CG-024` only inspected the filter argument, missing the classic mass-assignment shape** — `updateOne`/`updateMany`/`replaceOne`/`findOneAndUpdate`/`findOneAndReplace` all take `(filter, update, ...)`; passing the whole request object as the *update* document (`updateOne({_id}, req.body)`) is at least as common a real-world vulnerability as passing it as the filter, but was never checked. Now inspects both arguments for the update-family methods, using a new depth-aware `splitTopLevelArgs` helper instead of a naive `split(',')` (which breaks on an object-literal argument that has its own internal commas).
- **`CG-024`'s `$where` check wasn't actually scoped to the `$where` value** — it fired whenever `$where` appeared anywhere in the call text *and* the call had *any* dynamic (template/concat) argument anywhere, so an unrelated dynamic sibling field (`find({$where: "static"}, {note: \`hi ${x}\`})`) was flagged as if `$where` itself were dynamically built. Now locates the `$where` key's value position and requires the dynamic fragment to start exactly there.
- **`CG-025`'s default (JS/TS) config used exact receiver equality**, so it never matched the chained-call fix above — `res.status(302).redirect(url)` resolves to `object: "res.status(302)"`, which `=== "res"` rejects. Switched to `.includes()`, matching the substring-match convention already used by every other per-language receiver check in the codebase (`data.ts`, `ssrf.ts`).
- **Duplicated "known user-input source" regexes across `path.ts`, `redirect.ts`, and `ssrf.ts`**, with an unscoped bare `argv` alternative that matched any variable literally named `argv` regardless of what it held. Consolidated into shared `USER_INPUT_JS_PY`/`USER_INPUT_GO`/`USER_INPUT_JAVA`/`USER_INPUT_PHP` exports in `parser/languages/shared.ts` (used by `path.ts` and `redirect.ts`; `ssrf.ts` keeps its own broader variant, which predates this diff and wasn't part of it), tightening `argv` to require index access (`argv[`) or `process.argv`.
- 8 new regression tests covering the parser fix, the receiver guard, and the three `CG-024`/`CG-025` precision fixes (410 tests total).
- **New rule: `CG-026` JWT Signature Bypass (CWE-347)**, covering JS/TS, Python, and PHP — a fifth new rule category. Flags JWT verification configured to accept the `"none"` algorithm (`algorithms: ['none']`), which disables signature checking entirely and lets an attacker forge any token by stripping the signature, and PyJWT's explicit `verify_signature: False` opt-out. The `none` match is scoped to the algorithms array but not to its first element, so an allow-list that accepts `none` alongside a real algorithm (`algorithms: ['HS256', 'none']`, equally exploitable) is caught too. PHP's `firebase/php-jwt` takes its algorithm allowlist as a positional array argument rather than a named option, so it's gated to `JWT::decode(...)` calls specifically rather than a bare `'none'` substring match. There is no lower-confidence variant here (unlike most other rules in this file) — no legitimate configuration ever allows the `none` algorithm or disables signature verification, so both patterns are unambiguous. Not ported to Go/Java: common JWT libraries in those languages don't have an equally clear, low-false-positive text signal for the same misconfiguration. 9 new tests plus fixture coverage across all 3 languages. Built-in rule count: 17 → 18.
- **New rule: `CG-070` XML External Entity / XXE (CWE-611)**, covering JS/TS, Python, Java, and PHP — a sixth new rule category. XXE is usually a matter of the *absence* of a hardening call, which Stage 1's flat node model can't observe, so each pattern is instead an explicit *opt-in to danger* — a flag or setting a developer had to write deliberately and that has no safe purpose — keeping false positives low: Python's lxml `resolve_entities=True` / `no_network=False`; Java's JAXP/SAX `setExpandEntityReferences(true)` and enabling the `load-external-dtd` / `external-general-entities` / `external-parameter-entities` features (the safe direction — setting those to `false`, or `disallow-doctype-decl` to `true` — is deliberately *not* matched, verified by a safe Java fixture that hardens both features); PHP's libxml `LIBXML_NOENT` flag and `libxml_disable_entity_loader(false)`; JS/TS's libxmljs `noent: true` parse option. Not ported to Go: `encoding/xml` doesn't resolve external entities by default, so there's no equivalent dangerous switch to flag. 11 new tests plus fixture coverage across all 5 languages. Built-in rule count: 18 → 19.

Caught by a focused code review of the CG-026/CG-070 additions above, before merge:

- **`CG-026` was not gated to actual JWT verify/decode calls**, producing two false positives: any PHP `JWT::decode` whose text contained an incidental `'none'` (e.g. `JWT::decode($jwt, $keys['none'], ['HS256'])` — a key-id lookup — while the algorithms are safely `['HS256']`), and any non-JWT JS/TS call taking an `algorithms` list with a `'none'` enum value (e.g. a compression/cipher `negotiate({ algorithms: ['aes256', 'none'] })`). The rule now requires the call to actually be a JWT verify/decode (`JWT::decode` for PHP, `.verify`/`.decode` elsewhere); the PHP `none` check additionally uses a negative lookbehind so a variable subscript like `$keys['none']` no longer counts as the algorithms array.
- **`CG-026` missed legacy PyJWT `verify=False`** (the pre-2.0 opt-out, superseded by `options={'verify_signature': False}`). Now covered — and safe to add precisely because the rule is gated to `decode` calls, so it can't be confused with requests' unrelated `verify=False` TLS-cert option (which is correctly CG-050's job, not CG-026's).
- **`CG-070` missed the `external-parameter-entities` SAX feature** (`http://xml.org/sax/features/external-parameter-entities` set to `true`), an equally XXE-dangerous enablement alongside the already-covered `load-external-dtd` / `external-general-entities`. Added to the Java pattern.
- 5 new regression tests covering all four fixes (435 tests total).
- **SARIF rule descriptors now carry CWE metadata**: each rule in the SARIF `tool.driver.rules` catalog gets a `properties.tags` entry (`security` plus `external/cwe/cwe-N`), a numeric `security-severity` (9.0/7.0/4.0/2.0 for critical/high/medium/low), and a MITRE `helpUri`. GitHub Code Scanning renders the CWE tag as a label on each alert and uses `security-severity` to rank them. The rule-ID→CWE map lives in one place (`src/rules/cwe.ts`) with a guard test asserting every built-in rule has an entry (and no stale entries remain), so a new rule can't ship without a CWE. Custom rules with no mapping still get the `security` tag and a severity score, just no CWE tag or `helpUri`. 4 new tests (439 total).

### Changed

- **Consolidated per-language `if/else-if` branching into per-language lookup tables** in `path.ts` (CG-030/CG-031), `auth.ts` (CG-021/CG-022), `data.ts` (CG-040), and `redos.ts` (CG-023) — each `check()` now does a single `MATCHERS[ctx.language] ?? DEFAULT` lookup instead of a repeated if/else-if chain, so supporting a new language is one map entry instead of edits scattered across multiple branches. Purely structural — all matching logic and array constants are unchanged (still hoisted to module scope, not recreated per call), confirmed behavior-identical by the full test suite and self-scan/fixture-directory finding counts before and after.
- **Deliberately left `xss.ts` (CG-010) and `injection.ts` (CG-001/002/003) with their existing `if/else` structure**: CG-010's JS/TS branch (arbitrary text-node substring matching) and Python/Java branch (`function_call` node + `extractCallInfo`) require fundamentally different node shapes, so forcing them into the same lookup table would blur the distinction rather than clarify it. `injection.ts`'s per-language branches each carry extra semantic steps beyond call-matching (two-step `Sprintf`/`String.format` detection, sanitization-pattern exclusions), so it isn't a simple "pick a predicate per language" shape — converting it would add regression risk for limited clarity gain. Documented in `docs/design/RULES.md` §4.1.
- **Consolidated four copies of the "extract a call's argument-list text" step into one shared `getArgumentsText()` helper** in `parser/languages/shared.ts`. `injection.ts` (`getArgsText`), `path.ts` (`firstArgIdentifier`), `redos.ts` (`extractArgsText`), and `xss.ts` (`hasOnlyStaticStringArg`) each re-implemented the same `findOuterArgumentsStart` + slice-to-last-`)` + trim dance before inspecting arguments; they now call the shared helper, so the backward paren-matching lives in exactly one place and the rule files no longer import `findOuterArgumentsStart` directly (only the language adapters do). Purely structural — behavior-identical, confirmed by the full test suite and unchanged self-scan/fixture finding counts.

### Removed

- **Dead `LanguageAdapter.mapNodeType()` method**, along with its per-adapter `NODE_TYPE_MAP` tables and 10 unit tests that exercised it in isolation. Every adapter implemented it and it was unit-tested, but the actual parsing pipeline (`parser/index.ts`) never calls it — node `type` is assigned directly via string literals at the call sites that build each node (`createNode('function_call', ...)`, etc.), not through this method. Confirmed via `grep` that it had zero callers outside its own tests, and confirmed removing it is safe for the published package: `dist/index.d.ts` exports no public types (`ai-codeguard` is a CLI-only package, `LanguageAdapter` was never part of any public API surface). Test count: 390 → 380 (10 tests removed, none replaced — there was nothing left to test).

## [0.4.0] — 2026-07-05

### Added

- **Deeper Go/Java rule coverage** — Go goes from 5 to 8 built-in rules, Java from 5 to 9:
  - `CG-021` Weak Cryptography — Go: any call into the `crypto/md5`/`sha1`/`des`/`rc4` packages (the package itself is the weak-algorithm signal, e.g. `md5.Sum(...)`, `des.NewCipher(...)`); Java: `MessageDigest`/`Cipher.getInstance(...)` called with a weak algorithm string (`"MD5"`, `"SHA-1"`, `"DES"`, `"RC4"`); `"SHA-256"` and other strong algorithms are not flagged
  - `CG-040` Sensitive Data Exposure — Go: `log`/`logrus`/`zap`/`zerolog` logging calls; Java: `logger`/`log`/`System.out`/`System.err` logging calls; both reuse the existing password/token/secret/PII text heuristics
  - `CG-041` Insecure Deserialization (Java only) — `readObject()` method calls, the classic `ObjectInputStream`/`XMLDecoder` gadget-chain vector; Go has no equivalently clean idiom and is not covered
  - `CG-050` Security Misconfiguration — Go: `tls.Config{InsecureSkipVerify: true}`; Java: Spring `.csrf().disable()`, `.allowedOrigins("*")`, `setSecure(false)`, `setHttpOnly(false)`
  - Parser-level addition: Go `composite_literal` nodes (struct literals like `&tls.Config{...}`) are now normalized into the AST so text-pattern rules like `CG-050` can see them — previously only call/template/concat/credential nodes were reachable, so a misconfiguration expressed as a struct literal (not wrapped in a function call) was invisible to Stage 1
  - 20 new tests across the four rules
- **PHP language support (MVP, 6th language)**: `tree-sitter-php` grammar bundled (`dist/tree-sitter/tree-sitter-php.wasm`), `.php` detection, a new `phpAdapter` (splits `->`/`::` receivers from callees, resolves the outer call of chained invocations the same way the Java adapter does), and 6 rules:
  - `CG-001` SQL Injection — bare `mysqli_query`/`pg_query`/etc., or `->query/exec/prepare` (PDO/mysqli objects) and `Class::query` (e.g. Laravel's `DB::query`) with concatenated or interpolated SQL. Unlike the other languages this requires actual concatenation/interpolation rather than a bare SQL-keyword sniff — PDO's idiomatic `$pdo->prepare("SELECT ... WHERE id = ?")` takes the query as its only argument (parameters bind via a later `execute([...])`), so keyword-sniffing a plain literal would flag that standard safe pattern on every use
  - `CG-002` Command Injection — `exec`/`system`/`popen` (already caught by the existing cross-language function lists) plus PHP-specific `shell_exec`/`passthru`/`proc_open`
  - `CG-003` Code Injection (eval) — PHP's `eval()` parses as an ordinary function call in the grammar, so it's covered by the existing `EVAL_FUNCTIONS` list with no new code
  - `CG-020` Hardcoded Credentials — PHP's `assignment_expression` is picked up by the same normalization branch already used for JS/TS, no PHP-specific code needed
  - `CG-030` Path Traversal — global file functions (`file_get_contents`, `file_put_contents`, `fopen`, `readfile`, `unlink`, `copy`, `rename`, `mkdir`, `rmdir`, `is_file`, `file_exists`) with concatenated or interpolated paths
  - `CG-060` SSRF — `curl_init`, the most common real-world PHP SSRF vector, with a concatenated or interpolated URL
  - Parser-level notes: every double-quoted PHP string parses as `encapsed_string` regardless of whether it interpolates a variable — only ones with an actual interpolation child are treated as dynamic (`template_string`), so a plain literal like `"SuperSecret123!"` isn't mistaken for dynamic content; PHP's concatenation operator is `.`, not `+`
  - `.php` added to the default `scan.include` glob and the `init` config template
  - 27 new tests (rule units, parser precision, `phpAdapter` unit tests) plus vulnerable/safe PHP fixtures (`injection.php` / `secure-code.php`)

### Fixed

- **Nested same-rule duplicate findings** — Stage 1's lack of dataflow analysis meant a rule could independently fire on both an outer call and a call nested inside it for the same underlying issue (e.g. `db.Query(fmt.Sprintf(...))` reported the `db.Query` call and the `fmt.Sprintf` call as two separate `CG-001` findings; nesting a Go `tls.Config` literal inside an `http.Transport` literal double-reported `CG-050`). `runRules()` now suppresses a finding when its location is fully contained within another finding of the same rule in the same file, keeping only the outer, more-contextual one. The genuinely separate two-step pattern (`query := fmt.Sprintf(...); db.Query(query)`, not nested) is unaffected and still reported once.
- 5 new tests covering the suppression and confirming the two-step pattern still reports
- **`CG-040` missed capitalized logger objects in JS/TS/Python** — the Go/Java branches added above lowercase the receiver before matching (`Logger.fatal(...)` → `logger`), but the pre-existing JS/TS/Python branch never did, so the same call pattern (e.g. `Logger.fatal("token=" + token)`) was silently missed in those languages while the equivalent Go/Java code was caught. The receiver is now matched case-insensitively there too. Bare call names deliberately stay case-sensitive — lowercasing them would make `new Error(...)` collide with the `error` entry in `LOG_FUNCTIONS` and misclassify ordinary error construction as a log call (caught by self-scanning `src/` after the initial fix and narrowed before commit).
- **`CG-002`'s Python-function matching wasn't gated to Python** — `isPyCmd` (matching bare `system`/`popen`/`call`/`run`/`Popen`/etc.) never checked `ctx.language`, so adding PHP to `CG-002`'s language list newly exposed any Go/PHP codebase to false positives from unrelated bare functions coincidentally named `call`/`run`/`Popen` with dynamic string content. `isPyCmd` is now gated to Python; PHP's `system`/`popen` (previously covered only as a side effect of the ungated check) are now matched explicitly via `CMD_FUNCTIONS_PHP` instead.
- 6 new tests: capitalized-logger-object detection, the `new Error(...)` false-positive guard, PHP `system`/`popen` detection, and Go/PHP guards against the cross-language false positive

### Changed

- Extracted the byte-for-byte-identical `findOuterArgumentsStart` helper (chained-call outer-argument-list resolution) out of `java.ts` and `php.ts` into a shared `src/parser/languages/shared.ts`, so a future fix to the paren-matching logic can't silently apply to one language's adapter and not the other's
- `runRules()`'s nested-duplicate suppression now groups findings by `ruleId` before doing the pairwise containment check, instead of comparing every finding in a file against every other finding — containment can only ever apply within the same rule, so this avoids O(n²) comparisons across unrelated rule IDs on files with many findings

Suite now at 335 tests (+1 opt-in skip); behavior verified unchanged (self-scan and fixture-directory finding counts identical before/after).
