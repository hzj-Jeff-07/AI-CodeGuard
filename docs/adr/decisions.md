# ADR-001: 采用两级管道架构

- **状态**: 已接受
- **日期**: 2026-04-10
- **决策者**: 项目团队

## 背景

代码安全扫描有两种主流方案：

1. **纯静态分析**（Semgrep / CodeQL）：基于规则的模式匹配，速度快但误报率高，无法理解代码语义
2. **纯 LLM 分析**：让 LLM 直接审计全部代码，准确率高但成本极高、速度慢

我们需要找到准确性和成本之间的平衡点。

## 决策

采用**两级管道架构（Two-Stage Pipeline）**：

- **Stage 1 - 静态预过滤**：先通过本地 Tree-sitter 归一化解析 + 规则引擎快速匹配，过滤掉大部分明显安全代码
- **Stage 2 - LLM 深度分析**：仅对剩余可疑代码调用 LLM 进行语义分析

## 理由

| 方案 | 准确率 | 成本 | 速度 |
|------|--------|------|------|
| 纯静态分析 | 中（高误报） | 极低 | 极快 |
| 纯 LLM | 高 | 极高 | 慢 |
| **两级管道** | **高** | **低** | **较快** |

- 真实项目中，大部分代码不需要昂贵的 LLM 分析
- 本地预过滤可显著减少 LLM 调用量
- 保留 LLM 对上下文与语义的理解能力，用于最终确认

## 当前实现状态说明

**截至 2026-04-11：**
- Stage 1 已在主流程中实现
- Stage 2 已接入 `scan()` 运行时路径
- `--dry-run` 会明确停在 Stage 1
- 默认扫描在命中 suspicious nodes 且配置可用时会进入 Stage 2
- `--fix` 会在 Stage 2 确认 finding 时填充修复建议

## 后果

- 需要维护规则层与 LLM 层两套能力
- 需要持续控制 Stage 1 的漏报 / 误报边界
- 文档与 CLI 说明必须区分：`--dry-run` 是 Stage 1-only，非 dry-run 才可能进入 Stage 2
- 预算截断或未命中 suspicious nodes 时，结果可能只包含 Stage 1 findings

# ADR-002: 选择 Tree-sitter 作为 AST 解析目标

- **状态**: 已接受
- **日期**: 2026-04-10
- **决策者**: 项目团队

## 背景

AST 解析是预过滤阶段精度提升的基础。候选方案包括 Babel + 各语言解析器、Tree-sitter、ANTLR 等。

## 决策

长期方案选择 **Tree-sitter**（通过 WASM binding 在 Node.js 中运行）作为统一 AST 解析基础。

## 理由

1. **统一接口**：所有语言使用同一套抽象
2. **性能好**：适合本地扫描场景
3. **生态成熟**：grammar 丰富，适合持续扩展
4. **容错解析能力强**：适合安全扫描中的非完美源码输入

## 当前实现状态说明

**截至 2026-04-11：**
- `web-tree-sitter` 与 JS / TSX / Python grammar 依赖已接入
- 当前主流程 parser 已切换到 Tree-sitter
- 规则层继续消费兼容归一化 AST，而不是直接依赖原生节点
- 构建产物需要携带 core wasm 与 grammar wasm 资产

## 后果

- 文档现在应把 Tree-sitter 视为当前主流程能力
- 后续 parser 演进仍需继续维护规则兼容层与测试基线

# ADR-003: LLM 提供商抽象与默认选型

- **状态**: 已接受
- **日期**: 2026-04-10
- **决策者**: 项目团队

## 背景

LLM 深度分析是本项目的重要差异化方向，需要保留多提供商扩展能力。

## 决策

1. 设计 Provider 抽象
2. 默认优先 Claude
3. 允许通过配置切换 OpenAI 等提供商

## 理由

- Claude 在代码理解和长上下文场景下表现稳定
- Provider 抽象有利于避免单一厂商绑定
- 成本与调用策略应通过统一层管理

## 当前实现状态说明

**截至 2026-04-11：**
- `@anthropic-ai/sdk` 与 `openai` 依赖已存在
- `llm` 配置结构已存在并被运行时消费
- Analyzer Provider 已接入 `scan()` 主流程
- 当前已支持 Claude / OpenAI provider 选择、并发控制、预算截断与可选 fix 建议

## 后果

- 文档必须区分“Provider 已接线”与“更高阶能力仍未完成”
- 成本估算依赖内置 pricing 映射，未知模型在启用预算控制时会报错
- 如后续新增 Provider，需要同时补齐 SDK 封装、测试替身与定价策略

# ADR-004: 选择 SARIF 作为主要输出格式

- **状态**: 已接受
- **日期**: 2026-04-10
- **决策者**: 项目团队

## 背景

扫描结果需要结构化输出，并兼容主流安全工具链。

## 决策

以 **SARIF v2.1.0** 作为标准化输出之一，同时支持 JSON 和 Text。

## 理由

1. **生态兼容性好**
2. **适合安全发现表达**
3. **便于后续对接 GitHub Security / 其他 SARIF 消费方**

## 当前实现状态说明

**截至 2026-04-11：**
- SARIF formatter 已实现
- JSON formatter 已实现
- Text formatter 已实现
- 但仓库中尚无 GitHub Action 打包 / 发布 / 上传链路

## 后果

- 可以把“支持导出 SARIF”视为已实现能力
- 不能把“已完成 GitHub Action 集成”视为已交付能力

# ADR-005: YAML 规则 DSL 设计

- **状态**: 已接受
- **日期**: 2026-04-10
- **决策者**: 项目团队

## 背景

项目长期希望支持内置规则和用户自定义规则，因此考虑采用 YAML 规则 DSL。

## 决策

长期设计上采用 YAML 规则 DSL 作为自定义规则表达方式。

## 理由

1. 可读性好
2. 便于安全规则维护
3. 易于与现有静态分析工具习惯对齐

## 当前实现状态说明

**截至 2026-04-12：**
- 当前内置规则仍为 TypeScript built-in rules
- `rules.custom` 已接入 `scan()` 运行时，可加载 YAML 文件或目录
- custom rules 会在运行时进行 schema 校验，并编译成与 built-in rules 相同的执行契约
- `rules validate/create/test` 已实现，其中 `rules test` 为 Stage 1-only smoke path
- custom rules 仍受当前归一化 AST 能力边界限制，不是完整语义/数据流规则系统

## 后果

- 文档可以把“runtime custom rule loading”与 `rules validate/create/test` 视为当前能力
- 但不能把当前 custom rules 描述为完整语义 / 数据流规则平台
- 如后续继续扩展，需要继续补充示例、错误提示、测试与路径语义说明
